你所在的电商平台要在双十一期间推出秒杀频道。运营已经确定了玩法：每天 6 个场次，每场 2 小时，约 200 个 SKU 参与，部分爆款商品库存只有几百件，但预约提醒人数超过 50 万。

市场部的要求很直接：**开抢瞬间不能崩，绝对不能超卖**（少卖可以接受，超卖会造成无法履约的客诉与赔付）。历史数据表明，开抢前 1 分钟活动页刷新流量会达到平时的 40 倍，开抢瞬间下单请求峰值预计 5 万 QPS，其中大量是脚本和黄牛流量。

现有基础设施：公司主站是常规的 Java 微服务架构（Spring Cloud + MySQL + Redis + RocketMQ），订单、库存、支付都是独立服务，日常订单量约 30 万单/天。秒杀系统可以新建独立服务，也可以在现有服务上扩展。

**你需要设计这个秒杀系统的完整架构**，至少要说清楚：

1. 流量如何分层拦截，多少流量会真正到达下单链路
2. 库存扣减方案：在哪里扣、怎么保证不超卖、怎么应对热点商品
3. 下单链路的同步/异步边界，以及每个异步环节失败后的数据一致性处理
4. 如何应对脚本与黄牛
5. 容量推算：给出各层需要的实例数量级和依据
6. 秒杀系统故障时如何避免影响主站交易
